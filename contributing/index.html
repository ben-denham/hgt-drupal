Contributing

If you're reading this contributing guide, thanks for taking the time to help out!

This document will cover a few basic things to help you get started writing awesome documentation!

Setting Up a Development Environment

In order to make it easier to submit a pull-request with your contribution later on, you should start by creating a fork of this repository on GitHub by creating a GitHub account (if you don't already have one) and clicking the "Fork" button in the top-right corner of the project's GitHub page.

Once you have created your fork, follow the instructions for cloning your forked repository onto your local machine.

The webpages for the guide are generated with the MkDocs tool based on the content of the Markdown files in the docs/ directory.

Once you have MkDocs installed, you can start an automatic build/watch/serve process by running the following command from the root directoy of the project:

mkdocs serve

You can then open up a local copy of the webpages at http://localhost:8000, and any changes you make to the Markdown files will be reflected when you refresh the webpages.

Project Structure

Each Markdown file in the docs/ directory should cover a single topic, and should be completely self-contained and not dependent on any other part of the guide (except where noted; see below).

Each topic should have the following sections:

  1. Title
  2. What's the Big Idea?
  3. This should briefly explain what the section is about, and why the reader would want to learn about this topic.
  4. This Section Assumes (optional)
  5. This is where you can state any topics the reader should already know about before reading the section. In most cases, the other topics should be covered previously in the guide.
  6. Any subsections relevant to explaining your topic (optional)
  7. This is optional, as the "Learning Resources" section may link to guides that already sufficiently cover the topic.
  8. Learning Resources
  9. Any (freely available) external resources where the reader can learn about the topic.
  10. For each resource, a description should be given that explains what the resource explains/teaches, and why the reader will benefit from that resource.
  11. Tips and Gotchas
  12. Any quick tips that will save the reader time, or gotchas that will prevent them from making the same mistakes that you've made. Think of what you wished you'd known sooner while learning about this topic.

Here is a template you can use for starting new sections:

# My Section Title

## What's the Big Idea?

@TODO

## This Section Assumes

1.

## Subsection

## Learning Resources

1. http://example.com
   * This resource teaches...

## Tips and Gotchas

1. Tip
2. Gotcha

Submitting Your Contributions

When you're ready to submit your work, please make a pull request from your fork to the main repository.

See GitHub's guide on making pull requests: https://help.github.com/articles/creating-a-pull-request.

Please be patient, as the maintainers of the repository will need to review your contribution before it is accepted, and make ask you to make amendments, or my decline your submission if they feel it does not contribute to achieving the goals of the guide.
